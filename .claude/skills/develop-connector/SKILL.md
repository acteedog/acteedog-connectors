# Connector Development Guide

This guide is the reference for implementing a new connector.
Follow the patterns established in `src/github-connector`, `src/jira-connector`, and `src/slack-connector`.

---

## User Responsibilities (AI Must Not Perform These)

The following tasks require human judgment or environment-specific tooling and must be performed by the user, not the AI:

- **Initial connector setup**: Creating the `src/{name}-connector/` directory and running `xtp plugin init` to scaffold the project. The `xtp.toml`, `prepare.sh`, and initial `pdk.gen.go` are generated at this step.
- **Schema changes and `pdk.gen.go` regeneration**: Edit `src/acteedog-connector-schema.yaml`, then run `make update-schema` from the repository root. This regenerates `pdk.gen.go` in every connector directory.
- **`catalog.json` updates**: After building the WASM binary, add the connector entry and generate the checksum using `catalog/scripts/generate-checksums.sh`.

The AI's role is to implement the Go source files inside an already-scaffolded connector directory.

## 1. Architecture Overview

Each connector is a **WASM plugin compiled with tinygo**.
The host application (Acteedog) loads it via the [extism](https://extism.org/) framework and calls exported functions with JSON as input/output.

### Network Communication Constraints

- HTTP requests must use `pdk.NewHTTPRequest` / `req.Send()`. Go's standard `net/http` is not available.
- Only hosts explicitly listed in the `allowed_hosts` field of `catalog.json` are accessible.
- If your connector needs to reach a new domain, add it to `allowed_hosts` in `catalog.json`.

### tinygo Constraints

- Parts of the Go standard library are unavailable (`os`, `net/http`, `database/sql`, etc.).
- Libraries that rely heavily on reflection may not work.
- Keep implementation within simple type conversions, string manipulation, and JSON processing.
- `time.Parse` / `time.Format` are available. Always normalize timestamps to UTC using `t.UTC()` to avoid accidental timezone-related bugs.

## 2. Directory Structure

Place a new connector under `src/{name}-connector/`.

```
src/{name}-connector/
├── main.go             # GetConfigSchema / TestConnection / validateConfig
├── fetch.go            # FetchActivities + fetchHTTPClient (PDK HTTP implementation)
├── enrichment.go       # EnrichContext + enrichHTTPClient (PDK HTTP implementation)
├── match_context.go    # MatchContext
├── logger.go           # Logger implementation wrapping pdk.Log
├── pdk.gen.go          # ★ Auto-generated — NEVER edit by hand
├── go.mod / go.sum
├── xtp.toml            # tinygo build configuration
├── prepare.sh          # Setup script run by xtp plugin init
├── dist/
│   └── plugin.wasm     # Build artifact
├── internal/
│   ├── core/
│   │   ├── id.go       # ConnectorID / API base URL constants / ResourceType constants / ID generators
│   │   ├── context.go  # Context type + ContextGenerator (Factory pattern)
│   │   └── logger.go   # Logger interface + NoopLogger (for tests)
│   ├── fetch/
│   │   ├── http.go     # HTTPClient interface (swappable in tests)
│   │   ├── config.go   # Config parsing and validation
│   │   └── fetch.go    # ActivityFetcher implementation
│   ├── enrich/
│   │   ├── http.go     # HTTPClient interface
│   │   ├── config.go   # enrichment_params parsing
│   │   └── enrich.go   # ContextEnricher implementation
│   └── match/          # Optional — omit if MatchContext needs no internal package
│       └── match.go    # URL → Context mapping logic
├── mock/               # Auto-generated by mockgen (regenerate with go generate)
│   ├── fetch/http.go
│   └── enrich/http.go
└── testdata/
    ├── events/         # JSON fixtures for FetchActivities tests
    └── enrichment/     # JSON fixtures for EnrichContext tests
```

### `pdk.gen.go`

`pdk.gen.go` is auto-generated. **Never edit it by hand.** See the [User Responsibilities](#user-responsibilities-ai-must-not-perform-these) section for how to regenerate it.

## 3. Five Exported Functions to Implement

`pdk.gen.go` provides the WASM export boilerplate. You need to implement the following five Go functions.

### 3.1 `GetConfigSchema() (ConfigSchema, error)`

Returns the JSON Schema for user-supplied connector settings.

- Fields prefixed with `credential_` are treated as credentials (tokens, passwords, etc.).
- Include all required fields in the `Required` slice.
- Use JSON Schema-compatible type strings: `"string"`, `"array"`, `"boolean"`, etc.

### 3.2 `TestConnection(input TestConnectionRequest) error`

Verifies that the supplied configuration can actually reach the external API.

- Use a minimal API call (e.g. `GET /user` or `GET /me`) to validate the token.
- Return distinct error messages for HTTP 401 (auth failure), 403 (insufficient permissions), and 5xx (server error).
- Return `nil` on success; return `fmt.Errorf("...")` on failure.

### 3.3 `FetchActivities(input FetchRequest) (FetchResponse, error)`

Fetches the user's activities for a given date and returns them as a list of `Activity` objects.

- `input.Params.TargetDate` is the target date (RFC3339 or `2006-01-02` format).
- Implement fallback parsing for both formats (see the existing `parseDateRange` helper).
- Every `Activity` must include a `Contexts` slice ordered from source → intermediate levels → leaf resource.
- Set the `Source` field of each `Activity` to the connector's `ConnectorID` constant.

### 3.4 `EnrichContext(input EnrichRequest) (EnrichResponse, error)`

Enriches an existing Context with detailed information fetched from the API.

- The parameters needed for enrichment are stored in `input.Context.Metadata["enrichment_params"]`.
- If `enrichment_params` is absent, skip enrichment and return the input Context unchanged (do not return an error).
- Branch on `ResourceType` to decide which resource to enrich.
- When writing back to `Context.Metadata`, add new fields without overwriting existing ones — especially `enrichment_params`.

### 3.5 `MatchContext(input MatchContextRequest) (MatchContextResponse, error)`

Takes a slice of URLs and returns the corresponding Context hierarchy for each one.

- URLs that do not match must still appear in the result with `Contexts: []Context{}` (do not return an error).
- If the feature is out of scope or not yet implemented, returning an empty slice for every URL is acceptable (Slack uses this pattern).
- When URL patterns alone are sufficient (no external API needed), put the regex logic in `internal/match/` (GitHub's pattern).
- When an API call is required (e.g. Jira), define a dedicated `HTTPClient` interface in `internal/match/http.go`.

## 4. Domain Model Design

### 4.1 Context Structure

A Context is a node in a resource hierarchy tree. The following rules must always hold:

- Every connector has exactly one top-level **source context** with `ParentId: ""`.
- The `Contexts` slice of each `Activity` must contain the **complete path** from source to the leaf resource.
- Each Context's `ParentId` must exactly match its parent Context's `Id`.

Example:
```
source        (ParentId: "")
  └─ repository   (ParentId: source.Id)
       └─ pull_request  (ParentId: repository.Id)
```

### 4.2 Context ID Naming Convention

**Critical: Always follow this convention to prevent ID collisions across connectors and resource types.**

```
{connectorID}:{resourceType}[:{identifier...}]
```

- `connectorID` — lowercase service name (e.g. `github`, `jira`, `slack`).
- `resourceType` — defined as a constant in `id.go` (e.g. `source`, `repository`, `issue`).
- Use `/` or `:` as separators in identifiers, but **if an identifier itself contains `:`, choose a different separator**.

**Examples:**

| Connector | ID Example |
|---|---|
| github | `github:source` |
| github | `github:repository:owner/repo` |
| github | `github:pull_request:owner/repo:42` |
| jira | `jira:project:10001:issue:10025` |
| slack | `slack:channel:C0123456` |
| slack | `slack:thread:C0123456:1765613134.990399` |

**Key rules:**
- The source context ID is always the fixed value `{connectorID}:source`.
- ID generation functions must be **deterministic** — the same resource always produces the same ID.
- The `connectorID` prefix is mandatory to prevent collisions between connectors that may reference the same external resource type.

### 4.3 Activity ID Naming Convention

```
{connectorID}:{eventID or unique identifier}
```

Always include the `connectorID` prefix. When a single resource can produce multiple activity types (e.g. Jira's `created` / `comment:{id}` / `status_changed:{id}`), encode the type information in the ID as well.

### 4.4 Designing `enrichment_params`

Embed the parameters that `EnrichContext` needs to call the right API into `Context.Metadata` at creation time.

```go
Metadata: map[string]any{
    "enrichment_params": map[string]any{
        // Parameters required by EnrichContext
        "repo": "owner/repo",
    },
}
```

- Always embed `enrichment_params` when creating Contexts in `FetchActivities` or `MatchContext`.
- The source context's `enrichment_params` may be an empty map `{}`.
- Never include credentials in `enrichment_params`; they come from `config`.

### 4.5 ResourceType Constants

Define these as constants in `internal/core/id.go`.

- `ResourceTypeSource = "source"` is shared across all connectors.
- Define additional types to match the service's concepts (e.g. `repository`, `project`, `channel`, `thread`, `issue`, `pull_request`).
- Use snake_case lowercase strings.

## 5. Internal Package Design Patterns

### 5.1 `internal/core/`

Defines the connector-specific domain objects.

- **`id.go`**: `ConnectorID` constant, API base URL constant, `ResourceType*` constants, and ID generator functions.
- **`context.go`**: `Context` struct (distinct from the `Context` in `pdk.gen.go`) and `ContextGenerator`. The generator is created via `NewContextGenerator()` and exposes `Create*Context()` factory methods for each resource type.
- **`logger.go`**: Copy the `Logger` interface and `NoopLogger` from any existing connector — the content is identical across all connectors.

### 5.2 `internal/fetch/`

Owns the activity-fetching logic.

- **`http.go`**: `HTTPClient` interface — always define as an interface so tests can inject a mock.
- **`config.go`**: Parses the raw connector config (`map[string]any`) and `targetDate` string into a typed `config` struct. Return an error if required fields are missing.
- **`fetch.go`**: `ActivityFetcher` struct with a `FetchActivities()` method. Handles pagination, date filtering, and event-to-Activity transformation.

### 5.3 `internal/enrich/`

Owns the context-enrichment logic.

- **`http.go`**: `HTTPClient` interface for enrichment — defined separately from the fetch interface.
- **`config.go`**: Parses `contextType`, credentials, and `enrichment_params` into a typed `config` struct.
- **`enrich.go`**: `ContextEnricher` struct with an `EnrichContext()` method. Branch on `contextType` to call the appropriate API.

### 5.4 `internal/match/`

Used when `MatchContext` can be implemented without an internal package, or when URL-pattern-only matching is sufficient.

- If an API call is needed, define a `HTTPClient` interface in `http.go`.
- Use named capture groups in regular expressions for readability (see GitHub's `patterns.go`).

### 5.5 Dependency Injection Pattern

Each `ActivityFetcher` / `ContextEnricher` in `internal/` accepts an `HTTPClient` interface via its constructor. The top-level `fetch.go` / `enrichment.go` passes in the PDK implementation; tests pass in a mock.

```
fetch.go
  └─ fetch.NewActivityFetcher(&fetchHTTPClient{}, config, ...)
                                     ↑
                        PDK implementation of HTTPClient interface
                        (tests pass mock/fetch/http.go instead)
```

## 6. Testing Strategy

- Only `internal/...` packages are tested (`make test` scans this path).
- The `main` package (`fetch.go`, `enrichment.go`, etc.) depends on the WASM runtime and is not unit-tested.

### Mocking HTTP Clients

The `HTTPClient` interface can be mocked in two ways:

- **Hand-written mock** (preferred when the interface is small): Implement the interface directly in a `mock/` file. This keeps external dependencies minimal and is the approach used by the Jira connector.
- **mockgen** (optional, for larger interfaces): Use `go.uber.org/mock/mockgen` to auto-generate mocks. Add it to the `tool` directive in `go.mod`. Only introduce this dependency if the hand-written approach becomes impractical.

Prefer the hand-written approach to keep the dependency footprint small.

### Test Data

Place external API responses as JSON files under `testdata/`:

- `testdata/events/` — one file per event type, used by `FetchActivities` tests.
- `testdata/enrichment/` — one file per resource type, used by `EnrichContext` tests.

### NoopLogger

Pass `core.NewNoopLogger()` in tests to suppress log output.

## 7. Registering in `catalog.json`

**`catalog.json` updates are performed by the user.** See the [User Responsibilities](#user-responsibilities-ai-must-not-perform-these) section.

For reference, a connector entry looks like this:

```json
{
  "id": "{connector-name}",
  "name": "Display Name",
  "description": "Description. Include required API scopes/permissions.",
  "latest_version": "0.1.0",
  "allowed_hosts": ["api.example.com"],
  "capabilities": {
    "activity": true
  },
  "versions": [
    {
      "version": "0.1.0",
      "min_acteedog_version": "0.2.0",
      "download_url": "https://raw.githubusercontent.com/acteedog/acteedog-connectors/refs/heads/main/catalog/connectors/{id}/0.1.0/plugin.wasm",
      "checksum": "sha256:..."
    }
  ]
}
```

The checksum is generated with `catalog/scripts/generate-checksums.sh`.

## 8. Implementation Notes and Pitfalls

### Parsing HTTP Responses

As a general rule, define typed structs for API responses and use `json.Unmarshal` into them (see the Jira connector's `jira_types.go` files). This makes the code explicit, type-safe, and easier to test.

Using `map[string]any` is acceptable only when the response schema is highly dynamic or has a large number of varying shapes (the GitHub connector uses this approach due to the many different event payload structures). Treat `map[string]any` as an exception, not the default.

### Timestamp Parsing

- `FetchRequest.Params.TargetDate` can arrive as RFC3339 (`2006-01-02T00:00:00Z`) or date-only (`2006-01-02`). Implement fallback parsing for both (see the existing `parseDateRange` helper).
- Always normalize timestamps to UTC with `t.UTC()`.
- Some services return non-RFC3339 timezone offsets (e.g. `+0900` without a colon). Implement fallback parsing across multiple formats in that case (see Jira's `ParseJiraTime()`).

### Pagination

There is no single pagination strategy — choose the approach that fits the API:

- **Page-by-page iteration** (GitHub): loop through pages, stopping early when results fall outside the target date range.
- **Single large request** (Jira): set a high `maxResults` (e.g. 5000) and fetch everything in one call, relying on a server-side date filter in the query.

Review the API documentation and pick the most appropriate strategy. Avoid fetching more data than necessary.

### Error Handling

- Distinguish HTTP error codes in error messages: 401 (auth failure), 403 (insufficient permissions), 5xx (server error).
- For non-fatal errors (e.g. a single malformed event), log with `logger.Warn`, skip that item, and continue processing the rest.
- When `enrichment_params` is absent in `EnrichContext`, treat it as a no-op — return the input Context unchanged rather than an error.

### MatchContext Implementation Strategies

- **URL patterns are sufficient** (GitHub): implement regex logic in `internal/match/`; no external API call needed.
- **API call required** (Jira): define a `HTTPClient` interface in `internal/match/http.go` and fetch resource info from the API to build the Context hierarchy.
- **Out of scope / deferred** (Slack): return an empty slice for all URLs — `MatchContext` must always return without error.

### ConnectorID Uniqueness

- `ConnectorID` must be a lowercase alphabetic string that does not collide with any existing connector.
- Existing values: `github`, `jira`, `slack`.

### Go Module Name

- Use `{connector-name}-connector` as the module name (e.g. `module github-connector`).
- Keep external dependencies minimal and always verify the build works with tinygo.
